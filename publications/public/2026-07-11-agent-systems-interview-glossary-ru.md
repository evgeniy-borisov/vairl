*Серия «[Инженер агентных систем](/vairl/blog/2026/07/10/agent-systems-interview-ru/)» · глоссарий терминов*

Краткая таблица технических слов, сокращений и принятого сленга из блиц-опроса и design-кейсов серии.

## Платформа и роль

| Термин | Значение |
|--------|----------|
| **agent platform** | Платформа, где LLM, tools, память, оркестратор и evaluation работают как единая система. |
| **agentic pipeline** | Пайплайн с рассуждением, tool use и управлением состоянием между шагами. |
| **NL → pipeline** | Синтез исполняемого графа из текстовой постановки задачи на естественном языке. |
| **three contours** | Три контура платформы: генератор пайплайнов, DAG-оркестратор, эволюционный движок. |
| **pipeline generator** | Компонент, который из NL-описания собирает план, IR и исполняемый DAG. |
| **orchestrator / оркестратор** | Исполняет DAG: планирует узлы, собирает результаты, управляет retry и отменой. |
| **evolutionary engine** | Автоматически улучшает пайплайны по метрикам через поколения кандидатов. |
| **tool use layer** | Слой вызова внешних инструментов: registry, policy, sandbox, схемы IO. |
| **evaluation / benchmarking** | Оценка качества, стоимости и latency кандидатов на фиксированном наборе задач. |
| **trade-offs** | Компромиссы между latency, стоимостью, качеством, сложностью и надёжностью. |

## Python, asyncio и инженерия

| Термин | Значение |
|--------|----------|
| **asyncio** | Стандартная библиотека кооперативной конкурентности через корутины и event loop. |
| **coroutine / корутина** | Функция `async def`; выполняется только после `await` или `create_task`. |
| **event loop** | Планировщик asyncio, переключающий корутины в точках `await`. |
| **Task / TaskGroup** | Запущенная корутина; `TaskGroup` связывает жизненный цикл дочерних задач в одном блоке. |
| **asyncio.gather** | Параллельный запуск нескольких awaitable и сбор результатов. |
| **asyncio.to_thread** | Выполнение блокирующего sync-кода в thread pool без остановки event loop. |
| **GIL** | Global Interpreter Lock — в CPython только один поток исполняет bytecode одновременно. |
| **I/O-bound / CPU-bound** | Нагрузка, ограниченная ожиданием внешнего мира или вычислениями на CPU. |
| **backpressure** | Замедление upstream, когда downstream не успевает обрабатывать новые задачи. |
| **Pydantic** | Runtime-валидация, парсинг и нормализация схем данных на API- и tool-границах. |
| **dataclass** | Декоратор для структур данных: конфиги, DTO, события с минимальным бойлерплейтом. |
| **Protocol** | Структурная типизация поведения без наследования — «порт» к LLM или storage. |
| **TypedDict** | Типизация словарных JSON-пayload без полноценной runtime-валидации. |
| **DTO** | Data Transfer Object — объект передачи данных между слоями системы. |
| **payload** | Полезная нагрузка сообщения или API-запроса: JSON, аргументы tool call, тело HTTP. |
| **contextvars** | Изолированный между async-задачами контекст: trace id, tenant id, deadline. |
| **trace id / run id** | Сквозной идентификатор запроса или одного прогона агентного пайплайна. |
| **idempotency / idempotency key** | Повтор операции без дублирования побочных эффектов; ключ для безопасного retry. |
| **circuit breaker** | Блокировка вызовов нестабильного сервиса после серии сбоев. |
| **exponential backoff + jitter** | Растущая пауза между retry со случайной добавкой против синхронных всплесков. |

## Docker и инфраструктура

| Термин | Значение |
|--------|----------|
| **container / контейнер** | Изолированный процесс с отдельной ФС, сетью и лимитами ресурсов. |
| **Docker image** | Неизменяемый шаблон для запуска контейнеров. |
| **multi-stage build** | Сборка артефакта в промежуточном слое, в финальный образ попадает только нужное. |
| **digest** | Криптографический хеш образа; фиксирует точное содержимое вместо плавающего тега. |
| **healthcheck / readiness / liveness** | Проверки «жив ли процесс» и «готов ли принимать трафик». |
| **graceful shutdown** | Завершение текущих задач и закрытие соединений перед остановкой. |
| **rolling / blue-green / canary** | Стратегии выката без даунтайма или с постепенным переключением трафика. |
| **rollback** | Откат к предыдущей стабильной версии образа или конфигурации. |
| **sandbox** | Изолированная среда исполнения с лимитами сети, ФС, CPU и памяти. |
| **blast radius** | Область ущерба при сбое или компрометации компонента. |
| **least privilege** | Минимально необходимые права для процесса или инструмента. |
| **DLQ** | Dead-letter queue — очередь для сообщений, исчерпавших retry или не прошедших валидацию. |
| **at-least-once delivery** | Гарантия доставки ≥1 раза; возможны дубликаты, нужна идемпотентность. |
| **reproducibility** | Воспроизводимость результатов при фиксированных версиях кода, образов и seed. |

## Структуры данных и графы

| Термин | Значение |
|--------|----------|
| **hash table / хэш-таблица** | Структура с доступом по ключу через хэш-функцию; средняя сложность O(1). |
| **collision / load factor** | Коллизия ключей в одном бакете; заполненность таблицы, влияющая на коллизии. |
| **DAG** | Directed Acyclic Graph — ориентированный ациклический граф шагов пайплайна. |
| **topological sort** | Линейный порядок вершин DAG, где зависимости идут раньше зависимых. |
| **Kahn's algorithm** | Топосортировка через удаление вершин с нулевым in-degree. |
| **cycle detection** | Проверка, что граф не содержит циклов; обязательна перед запуском DAG. |
| **critical path** | Путь с максимальной суммарной длительностью; нижняя граница latency run. |
| **adjacency list** | Список соседей для каждой вершины; эффективен для разреженных графов. |
| **BFS / DFS** | Обход графа в ширину или глубину; базовые алгоритмы для DAG и поиска. |
| **priority queue / heapq** | Очередь с извлечением по приоритету за O(log n). |
| **LRU cache** | Least Recently Used — вытеснение давно не использованных элементов из кеша. |
| **cache stampede / single-flight** | Лавина запросов при cache miss; один запрос на miss, остальные ждут. |
| **consistent hashing / sharding** | Распределение ключей по кольцу или узлам с минимальной ребалансировкой. |
| **deduplication** | Атомарное резервирование детерминированного ключа задачи против дублей. |
| **state store** | Хранилище метаданных выполнения узлов: статус, тайминги, артефакты. |

## ML, метрики и evaluation

| Термин | Значение |
|--------|----------|
| **precision / recall / F1** | Точность, полнота и их гармоническое средство для бинарной классификации. |
| **ROC-AUC / PR-AUC** | Площадь под ROC или precision-recall кривой; PR-AUC лучше при редком классе. |
| **confusion matrix** | Матрица TP/FP/TN/FN для анализа типов ошибок. |
| **calibration** | Соответствие предсказанных вероятностей реальной частоте событий. |
| **bias-variance trade-off** | Баланс недообучения и переобучения модели. |
| **data leakage** | Утечка информации из test в train; завышает offline-метрики. |
| **holdout set / cross-validation** | Отложенная выборка или k-fold для честной оценки. |
| **data drift / concept drift** | Изменение распределения входов или связи «вход → правильный ответ». |
| **Pareto front / Pareto dominance** | Множество решений без доминирования по всем целям одновременно. |
| **multi-objective metric** | Одновременная оценка качества, стоимости, latency и надёжности. |
| **task success** | Доля успешно решённых задач агентом end-to-end. |
| **token cost** | Стоимость прогона в токенах LLM. |
| **offline / online evaluation** | Оценка на benchmark vs наблюдение реального поведения в проде. |
| **LLM-as-a-judge** | Оценка ответов другой LLM по rubric; требует калибровки. |
| **ablation study** | Исключение одного компонента для измерения его вклада. |
| **baseline** | Простой рабочий эталон для сравнения более сложных кандидатов. |
| **Recall@k / MRR / nDCG** | Метрики retrieval: доля релевантных в top-k, ранг первого hit, качество ранжирования. |
| **guardrail / release gate** | Пороговые ограничения и условия допуска к релизу или canary. |
| **benchmark suite / standardized runner** | Набор задач и единый запускатель кандидатов в одинаковой среде. |
| **benchmark contamination** | Попадание тестовых примеров в обучение или промпты модели. |

## LLM, трансформеры и RAG

| Термин | Значение |
|--------|----------|
| **LLM** | Large Language Model — большая языковая модель. |
| **Transformer / self-attention** | Архитектура на attention; каждая позиция взаимодействует со всеми в контексте. |
| **context window** | Максимальная длина контекста модели; не заменяет retrieval. |
| **KV cache** | Кеш key/value обработанных токенов для ускорения autoregressive decode. |
| **prefill / decode** | Параллельная обработка prompt vs последовательная генерация ответа. |
| **temperature / top-k / top-p** | Параметры стохастичности генерации: масштаб логитов, nucleus sampling. |
| **hallucination** | Уверенная генерация недостоверной информации без опоры в данных. |
| **SFT / RLHF** | Supervised Fine-Tuning и обучение с человеческой обратной связью. |
| **function calling / tool calling** | Формальный контракт вызова внешних инструментов через JSON-схему. |
| **structured output** | Генерация по схеме для передачи в код, БД или следующий шаг DAG. |
| **repair-prompt** | Повторный промпт для исправления невалидного JSON или схемы. |
| **prompt injection** | Попытка через данные переопределить системные инструкции модели. |
| **system prompt** | Доверенные правила системы, отделённые от пользовательских данных. |
| **CoT / Chain-of-Thought** | Пошаговое рассуждение модели перед финальным ответом. |
| **RAG** | Retrieval-Augmented Generation — дополнение генерации извлечённым контекстом. |
| **chunking / overlap** | Разбиение документов на фрагменты с перекрытием на границах. |
| **embedding / vector search** | Векторное представление текста и поиск ближайших соседей в индексе. |
| **BM25 / hybrid search** | Лексический sparse-поиск и его комбинация с dense retrieval. |
| **reranker** | Переранжирование top-k кандидатов retrieval для повышения релевантности. |
| **grounding / citation** | Привязка утверждений к источникам и явные ссылки в ответе. |
| **model routing** | Направление простых задач на дешёвую модель, сложных — на мощную. |
| **streaming** | Потоковая отдача токенов; снижает воспринимаемую задержку. |

## Генератор пайплайнов и DAG-оркестратор

| Термин | Значение |
|--------|----------|
| **intent parser** | Разбор NL-задачи на намерения: сбор данных, анализ, синтез. |
| **typed IR** | Промежуточное представление с input/output schema и tool requirements. |
| **DAG compiler** | Компиляция IR в исполняемый DAG с оценкой cost и latency. |
| **compile-time validation** | Статическая проверка графа до запуска: типы, инструменты, отсутствие циклов. |
| **schema mismatch** | Несовпадение полей между выходом одного шага и входом другого. |
| **repair-loop / contract diff** | Цикл исправления плана LLM по машиночитаемому diff схем. |
| **subgraph patch** | Локальная правка затронутого подграфа без полной регенерации плана. |
| **fallback template** | Шаблонный пайплайн при неудаче repair после N попыток. |
| **NodeSpec / EdgeSpec** | Спецификация узла и ребра зависимости в оркестраторе. |
| **ready queue** | Очередь узлов с нулевым in-degree, готовых к запуску. |
| **worker pool** | Пул исполнителей узлов с лимитами ресурсов и concurrency. |
| **run report** | Отчёт прогона: артефакты, тайминги, причинно-следственный trace. |
| **hard fail / soft fail** | Ошибка, останавливающая весь run, vs degraded-ветка без полной остановки. |
| **partial failure / degraded mode** | Часть узлов падает, но run продолжается с пониженным качеством. |
| **retry policy** | max_attempts, backoff и классификация retryable vs terminal errors. |
| **fan-out / fan-in** | Параллельный запуск независимых ветвей и сбор их результатов. |
| **telemetry** | Метрики и события оркестратора для анализа, отладки и эволюции. |

## Эволюционный поиск и agentic-паттерны

| Термин | Значение |
|--------|----------|
| **genome** | Кодировка кандидата: граф, промпты, tools, decoding-параметры. |
| **population / seed population** | Набор кандидатов поколения; старт из шаблонов и исторически сильных решений. |
| **crossover / mutation** | Скрещивание двух кандидатов и изменение потомка. |
| **LLM mutation** | Использование LLM как оператора мутаций: промпт, decomposition, tool set. |
| **MAP-Elites** | Алгоритм качественного разнообразия: лучший кандидат в каждой ячейке descriptor space. |
| **behavioral descriptors** | Оси поведения для MAP-Elites: глубина reasoning, степень tool usage. |
| **island model / migration** | Изолированные популяции с разными mutation policy и обменом топ-кандидатами. |
| **novelty archive / diversity metric** | Архив уникальных стратегий и мера разнообразия при отборе. |
| **premature convergence** | Преждевременная сходимость к одному локальному оптимуму. |
| **metric overfitting** | Оптимизация под конкретный benchmark, а не реальное качество. |
| **ReAct** | Reason + Act — чередование рассуждения и вызова инструментов. |
| **reflection / self-critique** | Самокритика и исправление ответа агентом с лимитом итераций. |
| **plan → execute → reflect** | Паттерн для high-risk задач: планирование, исполнение, рефлексия. |
| **supervisor / multi-agent debate** | Надзорный агент или спор нескольких агентов для критичных решений. |
| **fast path / complexity router** | Быстрый маршрут для low-risk vs сложный паттерн для high-risk задач. |
| **runaway cycle** | Бесконечный цикл refinement или tool-call без прогресса. |
| **strategy library** | Библиотека успешных шаблонов рефлексии и agentic-паттернов. |

## Память, tools и эксплуатация

| Термин | Значение |
|--------|----------|
| **short-term / long-term memory** | Ephemeral session memory vs persistent knowledge из прошлых задач. |
| **session memory / context builder** | Компактный state summary сессии и сборка prompt с token budget. |
| **write-back / retrieval policy** | Селективная запись артефактов в память и правила чтения из индекса. |
| **retrieval hit rate / groundedness** | Доля успешных retrieval и степень опоры ответа на источники. |
| **stale knowledge / index versioning** | Устаревшие факты в индексе; версионирование при обновлении документов. |
| **token budget** | Лимит токенов на контекст при сборке prompt. |
| **ToolRegistry** | Реестр инструментов с name, version, schemas и permissions. |
| **input_schema / output_schema** | JSON-схемы аргументов и результата инструмента. |
| **policy guard / allowlist** | Проверка доступа, rate limit и белый список разрешённых действий. |
| **sandbox executor** | Исполнение инструмента в изолированном раннере с timeout. |
| **error taxonomy** | Классификация ошибок: формат, аргументы, авторизация, таймаут. |
| **fallback path** | Кеш, альтернативный tool или упрощённый сценарий при сбое. |
| **contract tests** | Тесты соответствия инструмента заявленной схеме и поведению. |
| **MCP** | Model Context Protocol — стандарт подключения tools и контекста к агентам. |
| **SLI / SLO / error budget** | Индикатор, цель уровня сервиса и допустимая доля сбоев в окне. |
| **p95 / p99 latency** | 95-й/99-й перцентиль задержки; важнее среднего для SLA. |
| **correlation id / distributed traces** | Связка логов и сквозная трассировка вызовов между сервисами. |
| **PII / redaction / retention policy** | Персональные данные, маскирование в логах и политика хранения трасс. |

## Сокращения и жаргон

| Термин | Значение |
|--------|----------|
| **NL** | Natural Language — естественный язык в постановке задачи или промпте. |
| **API / HTTP / JSON** | Интерфейс сервиса, протокол запросов и формат обмена данными. |
| **SDK** | Software Development Kit — клиентская библиотека для вызова сервиса. |
| **IR** | Intermediate Representation — промежуточное представление между NL и DAG. |
| **IO** | Input/Output — входы и выходы шага, узла или инструмента. |
| **KPI** | Key Performance Indicator — ключевой показатель эффективности. |
| **SLA** | Service Level Agreement — договорённость об уровне сервиса с пользователем. |
| **PII** | Personally Identifiable Information — персональные идентифицируемые данные. |
| **TP/FP/TN/FN** | True/False Positive/Negative — ячейки confusion matrix. |
| **hot path** | Самый частый и критичный по latency путь исполнения запроса. |
| **happy path** | Сценарий без ошибок, отмен и деградаций. |
| **tail latency** | Задержки в «хвосте» распределения — p95, p99. |
| **blocking the loop** | Sync- или CPU-тяжёлый вызов внутри `async def`, останавливающий все корутины. |
| **висячая корутина** | Корутину создали, но не `await`-нули и не обернули в `Task`. |
| **promote to baseline** | Принятие успешной canary-версии как нового эталона для сравнения. |
