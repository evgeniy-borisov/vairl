## Summary

Outline логичен на верхнем уровне (возможности → архитектура → данные → технологии → фазы → агенты → математика → классификация → тренды), есть TL;DR и карта статьи — это сильная сторона. Но пост фактически склеен из **двух статей** (обзор wealth-платформы + методология классификации задач VAIRL) и делает **четыре прохода по одному и тому же жизненному циклу** из пяти фаз. Ритм ломают «стена таблиц» (53-строчная таблица моделей) и раздел классификации на 35% объёма с 11 таблицами подряд. Читатель Medium с высокой вероятностью бросит чтение на строке ~333 (полная таблица моделей) или ~500 (каталог задач). Нужны сокращения и слияние дублирующих разделов; вероятно — вынос классификации в отдельный пост.

## Findings

### Critical (must fix before publish)

- [ ] **«Стена таблиц» в «Математических моделях» (строки 331–398).** «Полная таблица» — 53 строки без единой разбивки, сразу за ней вторая таблица «Что используют чаще всего», которая её же и резюмирует. Это главная точка выхода читателя. Фикс: поменять местами — вести с дайджеста из 8 строк («Что используют чаще всего»), а полную таблицу либо свернуть в `<details>`, либо разбить на 4–5 тематических мини-таблиц с абзацем-связкой между ними, либо вынести в отдельный справочный материал.
- [ ] **Раздел «Классификация задач» (строки 444–689) — это вторая статья внутри первой.** 248 строк из 703 (~35%), 11 таблиц, 3 mermaid-диаграммы, YAML-листинг; аудитория другая (инженеры агентных систем VAIRL, а не «как банки строят wealth-платформы»). После него раздел «Куда движется индустрия» выглядит оторванным. Фикс: вынести в отдельный пост «Классификация задач wealth-платформы по трём осям VAIRL» с взаимными ссылками, а здесь оставить 1 подраздел на ~300 слов: три оси + таблица маршрутизации «Классификация → агент → pipeline» (строки 591–599) как мост к мультиагентной архитектуре. Если вынос не вариант — сократить минимум вдвое (убрать «Сквозные цепочки», task_record YAML, «Capability и eval», оставить оси + каталог + маршрутизацию).

### Major

- [ ] **Четыре прохода по пяти фазам цикла.** Таблица фаз (строки 63–69) → «Пять фаз: технологии» (211–283) → mermaid «Конвейер» (402–440) → «Каталог задач по фазам» (504–519). Фикс: раздел «Как это складывается в конвейер» целиком слить с «Пять фаз: технологии» — mermaid-схема конвейера должна открывать этот раздел, а не повторять его через 100 строк; итоговый абзац (строка 440) станет выводом объединённого раздела.
- [ ] **«Технологический стек» (153–207) дублирует «Пять фаз: технологии» (211–283).** Стек перечисляет Data Lake, Feature Store, RAG, Kafka/CEP, TS forecasting, XAI — а следующий раздел раскладывает те же технологии по фазам строками «Технологии: …». Фикс: либо реорганизовать стек по фазам (одним объединённым разделом «Технологии по фазам цикла»), либо ужать стек до компактной таблицы-глоссария «слой → зачем → примеры решений» на ~150 слов и убрать строки «Технологии:» из раздела фаз.
- [ ] **Две дублирующие архитектурные схемы.** Mermaid в «Общей архитектуре» (77–109) уже показывает оркестратор + шесть агентов + Data Platform; mermaid в «Мультиагентной архитектуре» (303–321) повторяет то же с девятью агентами (и наборы агентов между схемами не совпадают: «Финансы-агент» есть только в первой). Фикс: первую схему упростить до трёх уровней (клиент → агент/оркестратор → Data Platform), детальную с девятью агентами оставить в разделе 6; выровнять состав агентов.
- [ ] **Заключение без next steps.** «Куда движется индустрия» даёт takeaway (3 тренда + финальный абзац), но нет ни одной ссылки «что читать дальше» — все ссылки на VAIRL-посты (U–S–Y, постановка задачи агента, генерация бенчмарков) закопаны внутри раздела классификации. Фикс: добавить в конец блок из 2–3 ссылок на связанные посты VAIRL (и на будущий пост про классификацию, если её выносить).

### Minor / polish

- [ ] **Customer 360 (115–149) раздроблён:** семь H3, под каждым 1–2 предложения — обратная проблема «стены»: чтение спотыкается о заголовки. Фикс: свернуть в один список с bold-лидами («**Доходы** — зарплата, дивиденды…») или одну таблицу «блок профиля → источник данных → пример».
- [ ] **Переходы между H2 — механические `---` без связок.** Особенно резки стыки «Мультиагентная архитектура → Математические модели» (325–329) и «Конвейер → Классификация» (у второго связка есть, у первого нет). Добавить по одному мостику-предложению: чем закончили, зачем следующий раздел.
- [ ] **Матрица L×D содержит две таблицы подряд** (543–561) — сжать до одной (размещение задач), а ячейки-примеры L×D описать абзацем. Актуально, если раздел остаётся в посте.
- [ ] **Карта статьи из 10 пунктов** после сокращений нужно синхронизировать с новым составом разделов (сейчас пункты 4/5 и 8 указывают на дублирующие разделы).
