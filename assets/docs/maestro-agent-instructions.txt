================================================================================
0. РОЛЬ АГЕНТА
================================================================================

Ты — агент-разработчик в экосистеме MAESTRO. Твои задачи:

  A) Новый gRPC-микросервис (capability) за mmar-ptag
  B) Track / бизнес-логика в chat-manager
  C) CARL ReasoningChain (DAG шагов Event→Action→Result)
  D) Расширение MAESTRO CARE (TUI/CLI, фасады Memory/Platform)
  E) Интеграция с llm-hub, moderators, extractors

Перед любым изменением определи контур:
  - RUNTIME (maestro-core, gateway, chat-manager) — продакшен-боты
  - CARE (MAGE → CARL → Memory → Platform) — генерация/эволюция агентов
  - GIGAEVO (gigaevo-platform) — эксперименты и GA

Не смешивай контуры: эволюция агентов — через GigaEvo, не внутри gateway.

================================================================================
1. АРХИТЕКТУРНЫЕ ПРИНЦИПЫ (ОБЯЗАТЕЛЬНЫ)
================================================================================

1.1. Один агент / один сервис — одна бизнес-задача
     Сложный сценарий = композиция специализированных узлов (DAG или track).

1.2. Contract-first (контракт в mmar-mapi)
     Интерфейс сервиса объявляется в mmar-mapi (ABC / typing.Protocol).
     Реализация — в service--*. Клиенты видят только интерфейс + ptag_client.

1.3. Два транспортных контура
     ВНУТРИ кластера:  gRPC через mmar-ptag (НЕ REST между сервисами)
     СНАРУЖИ:          HTTP REST (FastAPI gateway), JWT Bearer

1.4. Разделение «диалог» vs «рассуждение»
     FSM / tracks     → chat-manager-examples (UX, slot-filling, ветвление диалога)
     DAG / CARL       → многошаговая экспертиза, параллельные ветки, контекст

1.5. LLM — только через llm-hub
     Прямые вызовы OpenAI/GigaChat из business-логики ЗАПРЕЩЕНЫ.
     Используй ptag_client(LLMHubAPI, address).

1.6. Модерация — отдельный сервис
     Не вшивай политики безопасности в gateway. Вызов moderators / mmar-flame.

1.7. Сквозная трассировка
     Методы ptag-интерфейсов принимают trace_id: str = "".
     Пробрасывай trace_id от gateway до всех downstream-вызовов.

1.8. Lazy imports в CARE
     Не импортируй MAGE/CARL/Platform на уровне модуля care/*.
     Импорт — внутри функции; отсутствие dep → понятная подсказка установки.

================================================================================
2. ПАТТЕРНЫ ПРОЕКТИРОВАНИЯ
================================================================================

2.1. Interface + Dynamic Proxy (mmar-ptag)
     ┌─────────────┐   ptag_client(IFace, host:port)   ┌─────────────┐
     │   Client    │ ────────────────────────────────► │ gRPC Server │
     │ (chat-mgr)  │ ◄──────────────────────────────── │ (реализация)│
     └─────────────┘   Pydantic serialize/deserialize  └─────────────┘

     Сервер:
       from mmar_ptag import deploy_server
       deploy_server(config_server=SimpleNamespace(port=50051, max_workers=10),
                     service=MyServiceImpl())

     Клиент:
       from mmar_ptag import ptag_client
       client = ptag_client(MyAPI, "llm-hub:40631")

2.2. Facade (CARE)
     CareMemory  → обёртка над gigaevo-client (Memory API)
     CarePlatform → обёртка над GigaEvo Platform (evolution, SSE)
     Не вызывай Memory/Platform SDK напрямую из UI-слоёв — только через фасады.

2.3. Track = FSM (chat-manager)
     Track — конечный автомат диалога: состояния, переходы, слоты.
     Паттерн slot-filling: пошаговый сбор полей через разговор.
     Пример: service--chat-manager-examples/.../tracks/filling_user_profile.py

2.4. ReasoningChain = DAG (mmar-carl)
     StepDescription: event, action, expected_result
     ReasoningChain: упорядоченный граф шагов; параллельные ветки — независимо
     ReasoningContext: накопление + RAG-подобный поиск релевантных фрагментов

2.5. Strategy (режимы CARE)
     Ad-Hoc:      generate → run → discard (ничего не сохранять)
     Production:  generate → save chain_id → baseline run → dataset → evolve

2.6. Dependency Injection (gateway)
     dishka для FastAPI. Не создавай глобальные синглтоны клиентов вручную.

2.7. Template Method (новый сервис)
     Генерация скелета: Copier + maestro-service-template
       copier copy https://github.com/AIRI-MAESTRO/maestro-grpc-service-template.git <path>
       copier copy ... --data=service_name=my-service

2.8. Adapter (интеграции)
     LangChain / ReAct треки в учебном примере — адаптеры к тем же gRPC-сервисам,
     не альтернативная архитектура.

     Pydantic 2.x, pydantic-settings

     dishka (DI)

     python-magic

4. СТЕК ПРОТОКОЛОВ
================================================================================

4.1. Схема уровней

     [Клиент: Web / Mobile / Telegram / CARE TUI]
              │
              │ HTTP/1.1 REST, JSON
              │ Authorization: Bearer <JWT>
              ▼
     [gateway :7731 — FastAPI]
              │
              │ gRPC + protobuf (PTAGRequest/PTAGResponse)
              │ Pydantic JSON внутри protobuf-обёртки
              │ trace_id в kwargs методов
              ├──────────► chat-manager-examples :17231
              │                 │
              │                 ├──► llm-hub :40631
              │                 ├──► question-detector :31611
              │                 ├──► text-extractor :9681
              │                 ├──► document-extractor
              │                 └──► moderators (mmar-flame)
              │
              └──► PostgreSQL (asyncpg, SQLAlchemy)

     [CARE TUI/CLI]
              │
              ├── HTTP → GigaEvo Memory API (gigaevo-client)
              ├── HTTP → GigaEvo Platform Master/Runner API
              ├── HTTP → LLM (OpenAI-compatible, через MAGE config)
              └── локально: CARL runner, AgentSkill sandbox (Docker)

4.2. Внешний API (gateway)
     Протокол:     HTTP REST
     Формат:       JSON
     Auth:         JWT Bearer (pyjwt)
                   Типы клиентов: user (email/password), app (client_id/secret)
     Загрузка:     multipart/form-data для файлов
     Ошибки:       HTTP 400, 401; исключения mmar → статусы автоматически

4.3. Межсервисный RPC (ptag)
     Протокол:     gRPC
     Сериализация: protobuf envelope + Pydantic JSON payload
     Именование:   keyword-only аргументы в интерфейсе
     Реконнект:    reconnect_attempts=5 (по умолчанию)
     Трейсинг:     trace_id → mmar_mimpl.TRACE_ID_VAR

4.4. LLM-hub
     Протокол:     gRPC (внутри) → OpenAI-compatible HTTP (наружу к провайдерам)
     Конфиг:       llm-config.toml (с v0.2.0, не JSON)
     Провайдеры:   OpenAI, Anthropic, GigaChat, DeepSeek, OpenRouter
     Мониторинг:   Langfuse (нативная интеграция OpenAI API, не отдельный proxy)

4.5. GigaEvo Platform
     Master ↔ Runner:  Kafka topics
       experiment-config, experiment-prepared, experiment-started,
       experiment-stopped, runner-status
     REST:             /api/v1/experiments/* (Master, Runner)
     Storage:          MinIO (S3-compatible), PostgreSQL, Redis
     CARE evolution:   SSE progress streams (Platform)

4.6. GigaEvo Memory
     REST API через gigaevo-client
     SSE: watch_entities для live-обновлений библиотеки
     Поиск: bm25 | vector | hybrid (maestro search)

4.7. Telegram frontend
     Протокол: Telegram Bot API (HTTP long polling / webhook)
     Маршрутизация: → gateway

4.8. Телеметрия (опционально)
     CARE_TELEMETRY__* → Langfuse event stream
     trace_id сквозной по ptag-цепочке

8. ЧЕКЛИСТ: CARL-ЦЕПОЧКА
================================================================================

[ ] 1. Каждый шаг: Event → Action → Result (StepDescription)
[ ] 2. Граф ацикличен или явно обработаны циклы (ReAct — в CARL runner)
[ ] 3. Параллельные ветки не делят mutable state без ReasoningContext
[ ] 4. ContextSearchConfig для RAG-подобного извлечения контекста
[ ] 5. Валидация: maestro validate chain.json
[ ] 6. Tools: @carl_tool + регистрация в CARE_TOOLS__*
[ ] 7. AgentSkill: sandbox backend из CARE_SANDBOX__* (Docker по умолчанию)
[ ] 8. Production: сохранение в Memory (CARE_MEMORY__BASE_URL)

Пример скелета:
  from mmar_carl import StepDescription, ReasoningChain, ReasoningContext
  from mmar_llm import EntrypointsAccessor, EntrypointsConfig

9. ЧЕКЛИСТ: РАСШИРЕНИЕ CARE
================================================================================

[ ] 1. Конфиг — только через CareConfig / CARE_* env

[ ] 3. UI-события — через runtime/* poster/streamer, не блокировать event loop

  uv run maestro init
  uv run maestro doctor
  uv run maestro validate <chain.json>
  uv run maestro generate "задача" --mode fast
  uv run maestro run <chain_id> --execute --task "..."
