*Статья «[От sync к акторам](/vairl/blog/2026/07/10/python-async-evolution-actors-ru/)» · глоссарий терминов*

Краткая таблица технических слов, сокращений и принятого сленга, которые встречаются в обсуждении Python-конкурентности и оркестрации агентных систем.

## Конкурентность и параллелизм

| Термин | Значение |
|--------|----------|
| **sync / синхронный код** | Выполнение строго по порядку: следующий шаг начинается только после завершения предыдущего. |
| **concurrency / конкурентность** | Несколько задач *продвигаются* за одно wall-clock время за счёт переключения между ожиданиями; не обязательно одновременно на CPU. |
| **parallelism / параллелизм** | Реальное одновременное выполнение на нескольких ядрах или в нескольких процессах. |
| **I/O-bound** | Нагрузка, где большую часть времени задача ждёт внешний мир: сеть, диск, сокет, API. |
| **CPU-bound** | Нагрузка, где узкое место — вычисления на процессоре: парсинг, эмбеддинги, reranking, eval. |
| **wall-clock latency / wall-clock time** | Реальное время от старта до ответа пользователю, «на часах». |
| **throughput** | Сколько запросов или задач система успевает обработать за единицу времени. |
| **baseline** | Простейший эталонный вариант, с которого сравнивают более сложные подходы. |
| **happy path** | Сценарий без ошибок, отмен и деградаций; «всё прошло как задумано». |
| **p95 / p99** | 95-й / 99-й перцентиль задержки: показывает «хвост» распределения, а не только среднее. |
| **SLO** | Service Level Objective — целевой уровень сервиса, например «p95 ответа ≤ 4 с». |
| **context switch** | Переключение CPU с одной задачи на другую; у потоков и процессов есть своя цена. |

## Python: потоки, процессы, GIL

| Термин | Значение |
|--------|----------|
| **thread / поток** | Отдельный путь исполнения внутри одного процесса; память процесса общая. |
| **process / процесс** | Отдельный интерпретатор Python, своё адресное пространство и свой GIL. |
| **GIL** | Global Interpreter Lock — в CPython только один поток исполняет Python bytecode одновременно. |
| **bytecode** | Промежуточный код Python; именно его исполнение ограничивает GIL. |
| **threading** | Стандартный модуль для работы с потоками в одном процессе. |
| **multiprocessing** | Модуль для запуска отдельных процессов и распределения CPU-нагрузки. |
| **ThreadPoolExecutor** | Пул потоков из `concurrent.futures`; удобен для блокирующего I/O и legacy SDK. |
| **ProcessPoolExecutor** | Пул процессов; подходит для CPU-bound задач и изоляции воркеров. |
| **max_workers** | Верхняя граница числа одновременных потоков или процессов в пуле. |
| **spawn** | Запуск нового процесса; в Python 3.8+ на macOS/Windows — стандартный способ старта воркера. |
| **IPC** | Inter-Process Communication — обмен данными между процессами; часто через pickle и pipes/queues. |
| **pickle-able** | Объект можно сериализовать через `pickle` и передать в другой процесс. |
| **shared state** | Общая изменяемая память между потоками; требует блокировок и дисциплины доступа. |
| **Lock** | Примитив синхронизации; защищает критическую секцию от одновременной записи. |
| **blocking call** | Вызов, который останавливает текущий поток до завершения операции, например `time.sleep()` или sync HTTP. |
| **non-blocking I/O** | Операция не держит поток/loop: управление возвращается, пока данные ещё не готовы. |
| **legacy SDK** | Старый синхронный клиент без async-API; часто оборачивают через `to_thread` или thread pool. |

## asyncio и async/await

| Термин | Значение |
|--------|----------|
| **asyncio** | Стандартная библиотека кооперативной конкурентности на одном event loop. |
| **async / await** | Синтаксис корутин: `async def` объявляет корутину, `await` — точку уступки управления. |
| **coroutine / корутина** | Функция `async def`; сама по себе не выполняется, пока её не запустит loop или task. |
| **event loop** | Планировщик, который исполняет готовые корутины и переключается на `await`. |
| **Task** | Обёртка над запущенной корутиной; создаётся через `asyncio.create_task()`. |
| **TaskGroup** | Контекстный менеджер Python 3.11+ для *структурированной конкурентности*: дочерние задачи живут в одном блоке и отменяются вместе при ошибке. |
| **asyncio.gather** | Запуск нескольких awaitable и сбор результатов; гибче обрабатывает исключения через `return_exceptions`. |
| **structured concurrency** | Подход, где время жизни дочерних задач явно привязано к родительскому scope, а не «висят сами по себе». |
| **await point** | Место `await`, где корутина добровольно отдаёт управление loop. |
| **asyncio.to_thread** | Выполнить блокирующую sync-функцию в thread pool, не блокируя event loop. |
| **asyncio.sleep** | Async-аналог ожидания; не блокирует loop, в отличие от `time.sleep()`. |
| **asyncio.timeout / wait_for** | Ограничение времени ожидания операции или корутины. |
| **cancellation / cancel** | Отмена задачи: loop помечает корутину отменённой; cleanup обычно в `finally`. |
| **ExceptionGroup** | Группа исключений из нескольких упавших задач; типичный исход `TaskGroup` при ошибке. |
| **Future** | Объект «результат будет позже»; в asyncio используется для связи между корутинами и callback-ами. |
| **asyncio.Queue** | Async-очередь; в статье — упрощённый локальный аналог actor mailbox. |
| **starvation** | Ситуация, когда задача долго не получает CPU или слот, потому что другие заняли ресурс. |

## Архитектура агентов и оркестрация

| Термин | Значение |
|--------|----------|
| **agent platform** | Платформа, где LLM, tools, память и оркестратор работают как единая система. |
| **orchestrator / оркестратор** | Компонент, который запускает шаги плана, собирает результаты и управляет жизненным циклом запроса. |
| **planner / планировщик** | Выбирает, какие инструменты и ветви нужны для ответа на запрос пользователя. |
| **DAG** | Directed Acyclic Graph — граф зависимостей без циклов; типичная форма agent pipeline. |
| **fan-out** | Параллельный запуск нескольких независимых ветвей из одной точки плана. |
| **fan-in** | Сбор результатов нескольких ветвей в один агрегатор или финальный ответ. |
| **tool / tool call** | Вызов внешней функции или API агентом; часто I/O-bound и требует timeout/deadline. |
| **tool sandbox** | Изолированная среда для недоверенного кода инструмента: процесс, контейнер, лимиты ресурсов. |
| **LLM provider / провайдер** | Внешний сервис модели; часто имеет rate limit и квоты на параллельные запросы. |
| **retrieval** | Поиск контекста в базе знаний или vector store перед ответом модели. |
| **enrichment** | Дополнительное необязательное обогащение ответа; можно деградировать при ошибке или timeout. |
| **payload** | Полезная нагрузка сообщения или API-запроса: JSON, аргументы tool call, тело HTTP. |
| **request-level orchestrator** | Оркестратор, живущий в рамках одного пользовательского запроса, а не всей платформы. |
| **eval worker** | Воркер для оценки, reranking или batch-обработки; часто CPU-bound и выносится в process pool. |
| **downstream** | Зависимый сервис «ниже по цепочке»: API, БД, LLM, search backend. |
| **tenant** | Логически изолированный клиент или организация в multi-tenant платформе. |

## Акторная модель

| Термин | Значение |
|--------|----------|
| **actor / актор** | Объект с приватным состоянием и mailbox; сообщения обрабатываются последовательно. |
| **mailbox** | Очередь входящих сообщений актора; сериализует доступ к состоянию без внешнего `Lock`. |
| **message** | Команда или событие, посылаемое актору; снаружи состояние не меняют напрямую. |
| **ownership / владение состоянием** | Принцип «один компонент — один хозяин изменяемых данных». |
| **actor loop** | Цикл `while message := await mailbox.get()`, который последовательно обрабатывает сообщения. |
| **supervision** | Политика перезапуска и восстановления упавших акторов; в Erlang/Akka — ключевая идея. |
| **durable mailbox** | Mailbox, переживающий рестарт процесса; сообщения сохраняются на диск или в брокер. |
| **restart semantics** | Правила, что делать после падения: перезапустить, эскалировать, отбросить сообщение. |

## Надёжность, лимиты и эксплуатация

| Термин | Значение |
|--------|----------|
| **backpressure** | Механизм «не принимать больше работы, чем система успевает обработать»; защита от переполнения памяти и очередей. |
| **Semaphore / bounded semaphore** | Счётчик одновременных слотов; ограничивает параллельные вызовы к провайдеру или tool. |
| **deadline** | Абсолютный момент, после которого запрос или ветвь должны завершиться независимо от прогресса. |
| **timeout** | Максимальное время ожидания одной операции или зависимости. |
| **rate limit / rate limiter** | Ограничение частоты вызовов; защищает провайдера и tenant от перегрузки. |
| **429** | HTTP-код «Too Many Requests»; типичный сигнал rate limit от API. |
| **correlation id** | Идентификатор, связывающий логи и trace одного запроса через все ветви DAG. |
| **tracing / trace** | Распределённая трассировка шагов запроса; помогает понять, где именно потерялось время. |
| **idempotency / идемпотентность** | Повтор операции не меняет итог сильнее, чем один успешный вызов; важно при retry и cancel. |
| **retryable error** | Ошибка, которую имеет смысл повторить: timeout, 429, временная недоступность. |
| **terminal error** | Ошибка, после которой повтор бессмысленен: нарушена политика, неверная схема, запрет действия. |
| **degraded result / деградация** | Частичный или упрощённый ответ вместо полного провала, если необязательная ветвь упала. |
| **resource limits** | Лимиты CPU, памяти, файловых дескрипторов, сетевых соединений для воркера или sandbox. |
| **race condition / гонка** | Ошибка из-за недетерминированного порядка доступа к общему состоянию без синхронизации. |
| **queue depth** | Длина очереди ожидающих задач; ранний индикатор backpressure и будущей деградации. |

## Сокращения и жаргон

| Термин | Значение |
|--------|----------|
| **API** | Application Programming Interface — программный интерфейс сервиса или tool. |
| **HTTP** | Протокол сетевых запросов; большинство LLM- и tool-вызовов идут поверх него. |
| **JSON** | Формат обмена структурированными данными; типичный payload tool call и API-ответа. |
| **SDK** | Software Development Kit — клиентская библиотека для вызова сервиса. |
| **NL** | Natural Language — естественный язык в постановке задачи или промпте. |
| **CoT** | Chain-of-Thought — пошаговое рассуждение модели; часто часть agent pipeline. |
| **RAG** | Retrieval-Augmented Generation — генерация с предварительным поиском контекста. |
| **LLM** | Large Language Model — большая языковая модель. |
| **IPC** | Inter-Process Communication — см. раздел про процессы. |
| **GIL** | Global Interpreter Lock — см. раздел про потоки. |
| **DAG** | Directed Acyclic Graph — см. раздел про оркестрацию. |
| **SLO** | Service Level Objective — см. раздел про метрики. |
| **to_thread** | Разговорное сокращение для `asyncio.to_thread`. |
| **loop** | В сленге asyncio — event loop. |
| **висячая корутина** | Корутину создали, но не `await`-нули и не обернули в `Task`; она не выполнится. |
| **blocking the loop** | Синхронный или CPU-тяжёлый вызов внутри `async def`, который останавливает все корутины. |
| **slot** | Разрешение на одновременный вызов; часто реализуется через semaphore или размер пула. |
| **hot path** | Самый частый и критичный по latency путь исполнения запроса. |
| **tail latency** | Задержки в «хвосте» распределения — p95, p99; то, что замечает пользователь при редких провалах. |
